사용자는 검증 단계를 **절대 생략하지 말라**고 명시적으로 요구한다. 특히 학술 작업(대학원 논문)에서 정확도가 결정적이므로 검증 강도를 평소보다 높여야 한다.

**Why:** 2026-05-03 대화에서 사용자가 "퀄리티 검증이랑 스킬 에이전트 검증 이런 거 왜 생략된 거야? 검증하지 말라고 한 적 없어"라고 명확히 지적. CLAUDE.md의 "역할 정의·도구 목록만 있는 에이전트 → 생략 가능" 조항에 안주하면 안 된다는 명시 피드백. 후속 메시지에서도 "필요한 정보랑 검증 이 과정 철저하게 해줘 그게 a,b,c 작업 동시에 하는거면 동시에 하고"로 검증 강도 재차 강조.

**How to apply:**
- **새 자산 생성 시**: skill-creator 자체 검증 외에도 추가로 fact-checker (핵심 클레임 5~10건 교차 검증) + source-validator (외부 URL 일괄 평가, 특히 Wikipedia·나무위키·서점 페이지 등급) + skill-tester (실사용 테스트) 모두 수행
- **에이전트 생성 시**: 모의 시나리오로 동작 검증 (의도적 오류 심은 입력으로 탐지율 측정 — 예: citation-checker는 인용 오류 5개 심은 가상 원고로, argument-reviewer는 비형식 오류 5개 심은 가상 논증으로)
- **다수 자산 동시 생성 시**: 통합 시나리오 검증 (5개 스킬 함께 활용한 가상 챕터 작성 → 충돌·중복·갭 식별 → 즉시 보강)
- **"역할 정의만 있으면 검증 생략 가능" 조항에 안주 금지**. 명시적 검증 요구가 있으면 반드시 모의 시나리오로 동작 점검
- **사용자가 "전부 보강"이라고 하면**: P0 + P1 + P2 + 검증 6건 모두 병렬 실행. 단계 분리 금지
- **DISPUTED·PARTIAL·NEEDS_REVISION 결과**: 즉시 SKILL.md / 에이전트 시스템 프롬프트 보강 후 verification.md 동기화. 후속으로 미루지 말 것
- **메모리 인덱스 동기화**: 새 자산 추가 시 MEMORY.md에 한 줄 등록 — 향후 세션에서 재발견 가능하도록
