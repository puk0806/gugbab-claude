```shell
/plugin marketplace add puk0806/00_gugbab-claude
/plugin install gugbab-frontend@gugbab-claude
/plugin install gugbab-core@gugbab-claude
```

## 설계할 때 주의할 점 2가지

**1. 플러그인끼리 파일 공유가 안 됩니다.** 설치 시 플러그인 디렉토리가 캐시 위치로 **복사**되기 때문에 `../shared-utils` 같은 바깥 경로 참조가 깨집니다. 공식 문서는 symlink를 권장하는데, 이 레포는 이미 `scripts/gen-settings.js` 같은 생성 스크립트 전례가 있으니 **빌드 스크립트로 공통 자산(공통 훅·rules)을 각 플러그인 폴더에 복사/링크해 넣는 방식**이 현실적입니다. 공통 훅들은 `gugbab-core` 플러그인 하나로 분리하는 게 가장 깔끔하고요.

**2. 설치 단위는 플러그인이 최소 단위입니다.** 플러그인 안의 스킬 일부만 골라 설치하는 건 안 됩니다. 지금 project-install.sh에서 `JAVA_SKILLS_*` 필터로 하던 세밀한 선택은, 플러그인 세계에서는 **경계를 더 잘게 쪼개는 것**으로 해결해야 합니다 — 예: `gugbab-java-legacy` / `gugbab-java-modern`을 별도 플러그인으로. 반대로 그 정도 세밀함이 꼭 필요 없다면 카테고리 단위로 묶으면 됩니다.

## 제안 방향

- **1단계**: 자산을 `plugins/{core,frontend,java-backend,...}` 구조로 재배치 (또는 기존 `.claude/` 구조를 유지하고 생성 스크립트로 plugins/를 빌드)

## 중복이 생기는 두 경로

**1. 이름 충돌 자체는 시스템이 막아줍니다.** 플러그인 자산은 플러그인 이름으로 네임스페이스가 붙습니다(앞서 문서에서 확인한 `/quality-review-plugin:quality-review` 형태). 그래서 `gugbab-frontend`와 `gugbab-java-backend`에 같은 이름의 스킬이 있어도 **깨지지는 않습니다.**

**2. 진짜 문제는 "같은 내용이 두 번 로드되는 것"입니다.** 제가 앞서 말한 "공통 자산을 빌드 스크립트로 각 플러그인에 복사" 방식이 바로 이 함정이에요. 예를 들어 공통 훅을 frontend·java-backend 양쪽에 복사해 넣으면, 두 플러그인을 같이 설치한 프로젝트에서:

- **스킬/에이전트**: 같은 설명(description)이 컨텍스트에 두 번 올라감 → 토큰 낭비 + Claude가 어느 쪽을 호출할지 혼란
- **훅**: 같은 훅이 **두 번 실행됨** — 이건 낭비가 아니라 실제 오동작 (커밋 훅이 두 번 돌거나, 가드가 중복 차단)

## 해법: 단일 소유 원칙 (asset은 정확히 1개 플러그인에만)

복사 방식을 버리고, **모든 자산이 정확히 하나의 플러그인에만 속하도록** 경계를 긋는 겁니다:

```
plugins/
├── core/           ← 공통 훅·공통 커맨드·메타 에이전트 (전제 조건 플러그인)
├── frontend/       ← 프론트 전용 스킬·에이전트만
├── java-backend/   ← Java 전용만
└── research/       ← 리서치 전용만
```

- 두 카테고리에 걸치는 자산(예: pr-reviewer, build-error-resolver 같은 범용 에이전트)은 **core로 승격** — frontend에도 java에도 넣지 않음
- core는 "다른 플러그인의 전제 조건"으로 문서화. 플러그인 간 공식 의존성 선언 기능은 없으니, 각 플러그인 README에 "gugbab-core 설치 필요"를 명시하고 설치 안내를 core부터 하도록 함
- 빌드 스크립트가 필요하다면 복사가 아니라 **"한 자산이 두 곳에 존재하면 빌드 실패"** 검증기로 씁니다 — 중복을 만드는 도구가 아니라 막는 도구로

이렇게 하면 사용자가 몇 개를 조합해 설치하든 각 스킬·에이전트·훅은 항상 한 번만 로드됩니다. 설계할 때 할 일은 결국 하나예요: **현재 202종 스킬 + 에이전트들을 "전용 vs 공통"으로 분류하는 소유권 매트릭스**를 만드는 것. 이 분류 작업이 플러그인 전환의 실질적인 첫 단계가 될 겁니다.

진행을 원하시면 카테고리별 자산 목록을 훑어서 소유권 분류안(어떤 자산이 어느 플러그인에 들어가고, 뭐가 core로 승격되는지)을 표로 먼저 만들어 확인받겠습니다.

## 두 방식의 차이

**말씀하신 방식 (코어만 플러그인 + 나머지는 프로젝트별 생성)**
- 프로젝트마다 스킬·에이전트 파일을 복사해 넣는 지금의 project-install.sh 방식이 유지됨
- 문제: 스킬을 수정하면 **이미 배포된 모든 프로젝트에는 반영 안 됨** — 프로젝트마다 다시 복사해야 하고, 시간이 지나면 프로젝트별로 버전이 제각각(drift)이 됩니다. 202종 스킬 규모에서는 관리 지옥이에요.

**제가 제안한 방식 (전부 플러그인화 + 분할)**
```
gugbab-core          ← 공통 훅·범용 에이전트 (모든 프로젝트가 설치)
gugbab-frontend      ← 프론트 스킬·에이전트
gugbab-java-backend  ← Java 스킬·에이전트
gugbab-research      ← 리서치 에이전트
...
```
- 프론트 프로젝트: `core + frontend` 설치 / Java 프로젝트: `core + java-backend` 설치
- 자산 파일은 **이 레포 한 곳에만 존재**하고, 각 프로젝트는 참조만 함
- 스킬을 고치면 이 레포에 커밋 → 각 프로젝트에서 `/plugin marketplace update` 한 번이면 최신화

즉 "각자의 프로젝트에서 생성"이 아니라, **"각자의 프로젝트에서 필요한 플러그인 조합만 설치"**입니다. 어제 걱정하신 중복 문제는 파일을 프로젝트로 복사해서 푸는 게 아니라, 자산 하나가 정확히 한 플러그인에만 속하게 경계를 그어서 푸는 거고요.

프로젝트에서 직접 만드는 건 딱 하나만 남습니다: **그 프로젝트에만 의미 있는 자산**(특정 서비스 도메인 스킬 같은 것). 범용성이 생기면 그때 이 레포의 해당 플러그인으로 승격시키면 됩니다.

정리하면 역할 분담은 이렇게 됩니다:

| 위치 | 담는 것 |
|------|---------|
| 이 레포 (마켓플레이스) | 재사용 가능한 모든 스킬·에이전트·훅, 카테고리별 플러그인으로 분할 |
| 각 프로젝트 | 플러그인 조합 설치 설정(`enabledPlugins`) + 그 프로젝트 전용 자산만 |

좋은 질문이에요. 방금 project-install.sh를 확인해 보니 **health 템플릿은 사실 자기만의 자산이 없습니다** — 스킬 포함 규칙이 `react-spa|nextjs|health`로 완전히 동일하게 묶여 있어요(frontend 스킬 + SEO writing + claude-code-headless 예외). 즉 health는 "자산 카테고리"가 아니라 **frontend 자산 세트를 바라보는 이름표(조합)**일 뿐입니다.

이게 핵심 구분입니다. 템플릿에는 두 종류가 섞여 있어요:

## 확인 결과

**1. "모든 대화 내용을 프로젝트 메모리에 저장하고 있다" — 아닙니다.**

`memory/`에 저장되는 건 대화 전체가 아니라 **선별된 사실 파일들**(feedback_*, project_* 등 요약된 지식)뿐입니다. 방금 실제 디렉토리를 확인했습니다:

- `~/.claude/projects/<프로젝트-해시>/memory/` → 정리된 메모리 파일 26개만 존재 (이 레포는 y 모드라 repo `memory/`로 symlink)
- **대화 원문 전체**는 별도로 `~/.claude/projects/<프로젝트-해시>/*.jsonl` 트랜스크립트로 저장됩니다 — 현재 이 프로젝트에 25개 세션 파일 확인

**2. 좋은 소식: 원하시는 "모든 대화 로컬 보관"은 이미 강제로 되고 있습니다.**

JSONL 트랜스크립트 저장은 Claude Code **내장 동작**이라 y/N 선택과 무관하게 항상, 모든 프로젝트에서 자동으로 됩니다. 끌 수도 없어요. 즉 N 프로젝트에서도 모든 대화는 이미 `~/.claude/projects/<해시>/`에 빠짐없이 남습니다.

**3. 남는 갭은 하나** — JSONL은 기계용 포맷이라 사람이 읽기 어렵다는 것. 그래서 처음 얘기하신 export(markdown 변환)가 필요한 거고, 이걸 **선택 옵션 없이 항상 동작하는 훅(강제)**으로 만들면 요구사항이 완성됩니다.

한 가지 제언: export 저장 위치를 `memory/` **안**으로 하는 건 피하는 게 좋습니다. y 모드에서는 memory/가 repo로 symlink되고 memory-sync 훅이 전부 자동 커밋하므로, 대화 전문이 공용 repo에 올라가 버립니다("공용 관리는 안 되더라도"라고 하신 것과 충돌). 메모리 recall 인덱스도 오염되고요. 대신 **형제 폴더** `~/.claude/projects/<해시>/exports/`를 쓰면 로컬 전용 보관이 깨끗하게 보장됩니다.

이렇게 이해했습니다:

메모리 공유 y/N과 무관하게, 모든 프로젝트에서 모든 대화가 사람이 읽을 수 있는 형태로 `~/.claude/projects/<해시>/` 아래에 **강제로(옵션 없이)** 보관되게 하고 싶다.

작업 목록:

1. `scripts/export-transcript.js` 작성 — 세션 JSONL → markdown 변환기
