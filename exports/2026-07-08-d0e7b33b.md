## 2. 플러그인 설치 절차 피드백 (`feedback_plugin_install.md`, 2026-06-11경)

외부 플러그인을 설치할 때의 확정된 절차:

1. 프로젝트 scope `settings.json`에는 `enabledPlugins`만 추가 (`extraKnownMarketplaces`는 넣지 않음 — 팀원에게 설치 프롬프트 뜸)
2. 본인 머신은 user-scope(`~/.claude/settings.json`)에 선언 + `claude plugin marketplace add` → `claude plugin install`을 Bash로 직접 실행
3. `claude plugin list`로 `Status: enabled` 확인까지 해야 완료 — settings.json 선언만으로는 설치 안 되고, `/reload-plugins`도 자동 등록 안 해줌

요약하면: **레포 자체의 플러그인화는 훅 다이어트 완료 후 남은 후순위 과제로 아직 미착수**이고, 설치 워크플로우 쪽은 절차가 확정돼 있는 상태입니다.

### [2] 요청

메모리 관리체계가 이렇게 되고있다는데 메모리를 어떨떄 기록하는지 보고하고 어떻게 커밋 푸쉬하는지 보고해 1. 메모리 관리 체계 보고

어떻게 저장되나: 메모리는 repo의 memory/ 폴더에 git으로 추적됩니다. ~/.claude/projects/<해시>/memory는 이 폴더를 가리키는 symlink라서, 제가 메모리를 쓰면 곧바로 repo 파일이 됩니다.

언제 작성하나 (자동 아님, 제 판단): 모든 대화를 다 기억하는 게 아닙니다. 제가 중요하다고 판단한 것만 선별 요약해서 Write/Edit로 기록합니다 — 프로젝트 진행 상태·설계 결정(예: relay-types 파이프라인), 사용자 피드백/선호(예: "커밋 전 보고"), 참조 정보. MEMORY.md는 인덱스라서 매 세션 시작 때 로드되고, 필요하면 개별 파일을 열어봅니다. 대화 전체에 가까운 기록은 별도로 session-export 훅이 세션 종료 시 요약본(요청·응답·수정 파일 목록)을 exports/에 남깁니다 — 이것도 전문 그대로는 아니고 요약입니다.
