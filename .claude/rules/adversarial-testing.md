# 적대적 테스트 원칙 (Adversarial Testing)

테스트 코드(단위·통합·E2E)를 작성하거나 export 시 test 코드가 생성될 때 준수한다.
superpowers `test-driven-development` 스킬의 "edge case 커버"를 이 레포에서 **강제 수준**으로 구체화한 규칙이다.

핵심 원칙:
> **정상 흐름만 통과하는 테스트는 미완성이다. 악성 유저가 시도할 법한 공격과 이상 경로까지 테스트해야 한다.**
> **그리고 구현은 "테스트를 통과시키기 위한 값"을 반환해선 안 된다 — 실제 로직으로 통과해야 한다.**

---

## 1. 테스트 3계층 — 전부 필수

기능 하나의 테스트 스위트는 아래 3계층을 **모두** 포함해야 한다. 정상 케이스만 있으면 미완성이다.

| 계층 | 목적 | 예시 |
|------|------|------|
| **정상 (Happy path)** | 의도된 입력 → 기대 출력 | 유효한 로그인, 정상 주문 |
| **악성 유저 방어 (Adversarial)** | 공격·오남용 시도가 **차단**되는지 | 권한 우회 시도 → 403, SQL 인젝션 페이로드 → 거부 |
| **이상·경계 경로 (Edge/Malformed)** | 비정상·경계 입력에서 **깨지지 않는지** | 빈 값·null·초장문·특수문자·동시 요청·타임아웃 |

`adversarial-test-guard` 훅이 테스트 파일 내용에 악성/경계 계층 흔적이 없으면 차단한다.

---

## 2. 악성 유저 공격 시나리오 체크리스트

테스트 대상에 아래 표면이 있으면 해당 항목의 방어 테스트를 반드시 넣는다.

### 인증·인가
- [ ] 비로그인 상태로 보호 리소스 접근 → 401
- [ ] 권한 없는 유저가 타인 리소스 접근(IDOR) → 403
- [ ] 만료·위조·타 유저 토큰 재사용 → 거부
- [ ] 수직(관리자 기능)·수평(다른 유저) 권한 상승 시도 → 거부

### 입력 인젝션
- [ ] SQL / NoSQL 인젝션 페이로드 → 파라미터 바인딩으로 무력화
- [ ] XSS 페이로드(`<script>`, `onerror=`) → 이스케이프·거부
- [ ] 커맨드 인젝션 / 경로 조작(`../`, 절대경로) → 거부
- [ ] 대용량·초장문·유니코드·제어문자 입력 → 검증 후 거부

### 비즈니스 로직 오남용
- [ ] 음수·0·오버플로 수량, 가격·금액 클라이언트 조작 → 서버 재검증
- [ ] 중복 제출·재생(replay) → 멱등성/중복 차단
- [ ] 레이스 컨디션(동시 요청) → 잔고/재고 음수 방지
- [ ] 단계 건너뛰기(결제 없이 완료) → 상태 전이 검증
- [ ] rate limit / 무차별 대입 → 제한·잠금

### 파일·요청 위조
- [ ] 위장 확장자·초대용량·악성 파일 업로드 → 차단
- [ ] CSRF(교차 출처 요청) → 토큰 검증
- [ ] 위·변조된 요청 본문·헤더 → 서버 검증

> 대상에 없는 표면(예: 파일 업로드가 없는 순수 계산 함수)은 생략 가능하나, **이상·경계 경로(계층 3)는 어떤 경우에도 생략하지 않는다.**

## 3. 테스트 통과용 가짜 구현 금지 (A: hard block)

TDD의 "최소 구현(GREEN)"을 **테스트 기대값을 그대로 반환하는 것**으로 악용해선 안 된다.

```typescript
// ❌ 금지 — 파라미터를 무시하고 테스트가 기대하는 리터럴을 그대로 반환
function calcDiscount(price: number, rate: number): number {
  return 900; // 테스트가 toBe(900)이라서 박아넣음
}

// ✅ 실제 로직으로 통과
function calcDiscount(price: number, rate: number): number {
  return Math.round(price * (1 - rate));
}
```

`fake-impl-guard` 훅이 **파라미터를 하나도 사용하지 않으면서 테스트 기대 리터럴을 그대로 return하는** 구현을 차단한다.

**가드가 걸리면 대응 순서:**
1. 구현이 가짜다 → 실제 로직으로 다시 구현한다.
2. 테스트가 잘못됐다(기대값·설계 오류) → **테스트를 다시 짜고** 그에 맞는 실제 구현을 한다.
3. 파라미터 없는 상수 반환이 정당한 경우(설정값·버전 getter 등)에만 → 근거를 명시한 waiver 주석을 단다:
   `// fake-impl-guard: allow — <이유>`

> 가드를 피하려고 리터럴을 변수로 우회하거나 파라미터를 형식적으로 건드리는 것은 규칙의 정신 위반이다.

## 4. 적용 대상·범위

| 대상 | 강제 방식 |
|------|-----------|
| 단위·통합 테스트 파일(`*.test.*`, `*.spec.*`, `*_test.*`, `__tests__/`) | `adversarial-test-guard` — 악성/경계 마커 없으면 차단 |
| 소스 구현 파일 | `fake-impl-guard` — 파라미터 무시 + 테스트 기대 리터럴 반환 시 차단 |
| E2E 테스트(Playwright 등) | `qa-engineer` 에이전트가 적대적 E2E 시나리오를 **필수 산출물**로 생성 |

두 훅 모두 **개발 전용(dev 템플릿)** 훅이다 — `tdd-guard`·`test-fake-guard`와 함께 export된다.

---

## 5. 왜 필요한가

- 정상 경로만 검증된 코드는 **공격 표면이 그대로 노출**된 채 배포된다. 인젝션·IDOR·권한 상승은 happy-path 테스트로는 절대 잡히지 않는다.
- 테스트 기대값을 박아넣은 가짜 구현은 **커버리지 수치는 올라가지만 실제로는 아무것도 검증하지 못한다** — 최악의 기술 부채다.
- "나중에 보안 테스트 추가"는 대부분 오지 않는다. 테스트 작성 시점에 강제해야 한다.
