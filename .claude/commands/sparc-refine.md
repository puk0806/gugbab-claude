# /sparc-refine

SPARC 방법론으로 지정한 코드·모듈·기능을 단계별로 리팩터링한다.

인자: `$ARGUMENTS` — 리팩터링 대상 (파일 경로, 함수명, 기능명)

인자가 없으면 현재 컨텍스트에서 최근 수정된 파일을 대상으로 한다.

---

## SPARC 5단계

### S — Specification (명세)

**목표**: 리팩터링 범위와 목표를 명확히 정의

- 대상 코드의 현재 역할과 문제점 분석
- 리팩터링 후 달성할 목표 상태 정의
- 범위 밖 항목 명시 (건드리지 않을 것)
- 완료 기준(DoD) 작성

```
[S 완료]
대상: ...
문제: ...
목표: ...
범위 밖: ...
DoD: ...
→ P 단계로 진행할까요?
```

### P — Pseudocode (의사코드)

**목표**: 새 구조의 핵심 로직을 코드 없이 기술

- 함수/컴포넌트 시그니처 스케치
- 제어 흐름과 의존성 방향
- 엣지 케이스 처리 전략

```
[P 완료]
새 구조 스케치:
  ...
엣지 케이스:
  ...
→ A 단계로 진행할까요?
```

### A — Architecture (아키텍처)

**목표**: 영향받는 파일과 변경 계획 확정

- 수정/생성/삭제할 파일 목록
- 레이어 간 의존성 변화
- 인터페이스 변경이 있으면 호출처 목록

```
[A 완료]
수정 파일:
  - {파일}: {변경 내용}
삭제 파일:
  - {파일}
호출처 영향:
  - {파일}: {수정 필요 여부}
→ R 단계로 진행할까요?
```

### R — Refinement (구현)

**목표**: A에서 확정한 계획대로 실제 코드 수정

- 파일별 순서대로 진행
- 각 파일 완료 시 ✅ 표시
- 계획 이탈 발생 시 즉시 보고

```
[R 진행 중]
✅ {파일} 수정 완료
🔄 {파일} 진행 중...
```

### C — Completion (완료 검증)

**목표**: 리팩터링 결과와 DoD 대조

- S단계 DoD 항목별 달성 여부 확인
- 계획에 없던 변경 사항 목록
- 후속 작업 제안

```
[C 완료]
DoD 달성:
  ✅ ...
  ✅ ...
계획 이탈: (없음 / 있으면 명시)
후속 작업: ...
```

---

## 규칙

- 각 단계 전환 시 사용자 확인 필요
- R 단계에서 A 계획 범위를 벗어나야 하면 즉시 중단하고 보고
- 단계 진행 중 새 문제 발견 시 현 단계에서 처리하거나 S로 복귀
